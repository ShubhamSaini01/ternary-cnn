Optimization Journey: Talking Points with Evidence

Ternary ResNet-18, CIFAR-10, Intel i5-13400F (Raptor Lake, AVX2 + AVX-VNNI) — 79ms → 1.00ms (79×)

Evidence tags: VTUNE VTune uarch-exploration (collected April 2026, results in /tmp/vtune_*) JOURNAL From docs/optimization_journal.md (original development) BENCH Engine benchmark (median, 200-500 runs) PERF Linux perf record + objdump

Step 1: Naive FP32 AVX2 Baseline → 79ms (1T)

What: FP32 im2col + GEMM with AVX2 FMA. BN fused into conv weights at export.

JOURNAL 79.26ms 1T. 31× slower than ONNX INT8 Static.
Starting from scratch — no INT8, no VNNI. This is our measure-first baseline.

Step 2: Transposed dpbusd VNNI → 17ms (1T)

What: Dynamic INT8 quantize → transpose → dpbusd per (OC, spatial_pos) with horizontal reduction.

JOURNAL Latency: 79ms → 17ms (4.7×)
JOURNAL VTune: CPI=0.294, Port 6 at 54.6% — horizontal reduction shuffles saturating a single execution port.
First VNNI usage. Huge win from INT8, but VTune shows Port 6 saturated — the bottleneck is horizontal reduction shuffles, not dpbusd throughput.

Step 7: Spatial-Packed dpbusd — BREAKTHROUGH → 9.7ms (1T)

What: Restructured dpbusd: 8 spatial positions × 4 reduction elements per instruction. Weight broadcast to all lanes. No horizontal reduction.

JOURNAL Latency: 17ms → 9.7ms (1.75×)
JOURNAL VTune: CPI 0.294 → 0.208, Retiring 54% → 72%, Port 6 pressure reduced, Split Loads 6% → 1%.
The single biggest architectural insight. Eliminating horizontal reduction unlocked 1.75× — VTune Port 6 saturation was the direct evidence.

Steps 20–22: New Optimizations (VTune evidence from this session)

Step 20: MLAS Disassembly + 14-OC Unrolled GEMM → 1.89ms (6T)

Hypothesis

ONNX Runtime uses MLAS (not oneDNN) for INT8 on AVX-VNNI. VTune shows GEMM L1 Latency Dependency = 68% — the load-dpbusd-store chain on stack accumulators. Unrolling dpbusd with register-only accumulators should eliminate this.

What we did

  1. PERF Disassembled MLAS kernel via perf record + objdump — confirmed 6-OC × 16-N with 12 YMM register accumulators, vpdpbusds
  2. Discovered oneDNN has NO JIT INT8 for AVX-VNNI — DNNL_VERBOSE=1 showed ref_int8:any (7,400× slower than hand-written VNNI)
  3. BENCH Swept OC tile sizes 6/8/10/12/14/16 with fully unrolled dpbusd
  4. 14-OC selected as sweet spot: good register pressure + activation reuse

Why it worked despite worse CPI: GEMM wall time halved (0.95s → 0.46s). 14 unrolled dpbusd per kg iteration executes more useful VNNI work. CPI worsened because im2col/repack (still serial, higher CPI) became a larger fraction.

Step 21: Parallel im2col + Repack → 1.14ms (6T)

Hypothesis

Engine profile at 6T: im2col+repack = 1.11ms (56% of total) running in omp single. GEMM only 0.66ms. Parallelizing im2col across K-rows and repack across kg groups should scale to 6 threads.

What we did

Replaced #pragma omp single { im2col; repack; } with:

  1. omp for schedule(static) over K-rows for im2col — each row writes to independent output region
  2. omp barrier — ensure im2col done before repack
  3. omp for schedule(static) over kg groups for repack — each group reads 4 rows, writes unique destination

Step 22: Parallel Conv1 FP32 → 1.00ms (6T)

Hypothesis

Engine profile: Conv1 FP32+ReLU+quantize = 0.21ms at 6T (17%), running in omp single. 64 output channels are independent.

What we did

Replaced omp single { conv2d_fp32(); relu(); quantize(); } with:

  1. omp single for FP32 im2col (shared data, small)
  2. omp for schedule(static) over 64 OCs: fused GEMM + bias + ReLU + quantize-to-u8 in one pass

Remaining Bottlenecks

VTUNE vtune_final_6t
  1. OMP barriers (23%): libgomp spin-wait. ~57 barriers per inference. Fusing im2col+repack into 1 omp for could save ~0.1ms.
  2. GEMM L1 Dep (35–66%): 14-OC partially spills registers. Layout change for 2-jg tiling could help (~0.05ms).
  3. Memory Bound (22%): 6 threads competing for L2/L3 during parallel im2col.

Estimated headroom: ~0.1–0.15ms. Theoretical floor: ~0.85ms at 6T.
