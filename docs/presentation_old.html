Architecture: INT8-Resident Execution with Spatial-Packed dpbusd

Inference Pipeline

Input FP32 → Conv1 FP32 GEMM → BN + ReLU → Quantize to uint8 → For each block: uint8 → im2col → repack → dpbusd GEMM → fused ReLU + requant → uint8 → AvgPool → FC → logits

What Stays INT8

  • Activations between all ternary layers
  • im2col bytes (uint8 input, int8 weights)
  • dpbusd: u8 × i8 → i32 accumulation
  • Static quantization scales (calibrated offline)

Where FP32 Appears

  • First convolution (Conv1)
  • GEMM epilogue (i32 → fp32 → scale → u8)
  • Residual addition (dequant + add + requant)
  • Shortcut convolutions (originally FP32, later INT8)

Key Innovation: Spatial-Packed dpbusd

Before: Reduction-Major dpbusd

  • 32 MACs along reduction dimension
  • 1 spatial position per instruction
  • Requires horizontal reduce after each group
  • Port 6 utilization: 55%
  • CPI: 0.294

After: Spatial-Packed dpbusd

  • 8 spatial × 4 reduction per instruction
  • No horizontal reduce needed
  • CPI: 0.171

Combined with 16-OC Tiling

Each dpbusd instruction processes 8 spatial positions across 4 reduction elements, with 16 output channels tiled in the outer loop. This eliminates the costly horizontal reduction and maximizes instruction throughput.

What Didn't Work (and Why)

Ternary-Specific Compute Tricks

  • INT16 add/sub: Fewer MACs/instruction than dpbusd
  • sign_epi16: Requires unpacking, negation, extra shuffle
  • Sparse / zero skipping: Branch misprediction overhead exceeds savings (only 14% group-level sparsity)
  • FP32 add/sub: Wider registers but half the throughput of INT8 dpbusd

Three Attempts to Eliminate im2col+repack

  • Fused implicit GEMM: 58ms (address computation overhead)
  • Fused im2col → packed: 4.71ms (marginal, 3x more instructions)
  • Unrolled 9-pos: 4.58ms (2.2x more instructions)

8-OC Register Tiling

L1 thrashing — insufficient registers to hold 8-OC tile without spilling, causing cache pressure.

The ~21% im2col+repack+memcpy overhead is the primary remaining gap to ONNX INT8 Static.

4-bit nibble-packed dpbusd 8.24ms 93.64% 1.75x slower

4-bit sign+add

Without dpbusd, ternary weights require explicit sign/add operations. Each MAC takes multiple instructions instead of dpbusd's 32 MACs per instruction. Result: 7x slower.

4-bit nibble-packed dpbusd

Packing two 4-bit values into one byte halves the input width, but the unpack/repack overhead and reduced dynamic range cost 1.75x. Accuracy also drops by 0.82pp.

dpbusd processes 32 MACs per instruction in INT8. Any scheme that cannot use dpbusd natively will be slower.

Gap Analysis: Why ONNX INT8 Static is 2x Faster

Key Discovery: ONNX Runtime Uses MLAS, Not oneDNN

oneDNN has no JIT INT8 kernels for AVX-VNNI (Raptor Lake). It falls back to reference C code — 7,400× slower than hand-written VNNI. ONNX Runtime instead uses MLAS (Microsoft Linear Algebra Subroutines), which has hand-optimized AVX-VNNI kernels with 1,902 vpdpbusd instructions in the binary.

What MLAS Does Differently

  • NHWC layout: Eliminates im2col entirely
  • Register-only accumulators: 12 YMM regs (ymm4–15), no stack spills
  • 6-OC × 16-N tiling: 2 ymm loads shared across 6 output channels
  • vpdpbusds: Saturating variant, same throughput

Our Remaining Overhead

GEMM 73%
im2col+repack 16%
other 11%

The ~21% im2col+repack+memcpy overhead is time MLAS eliminates via NHWC layout. Additionally, our GEMM kernel itself is slower due to stack-based accumulators.

Ternary vs INT8: Where Ternary Wins and Loses

Where Ternary Wins

  • Model size: 3.56 MB vs 11.23 MB (3.2x smaller)
  • Memory bandwidth: Smaller weights = less data to move
  • Edge deployment: Fits in tighter memory budgets

Where Ternary Loses

  • Compute throughput: dpbusd processes any INT8 value at the same speed — ternary gets no compute advantage
  • Sparsity: Only 14% group-level sparsity, not enough for skip-based speedup
  • LUT: Output channels too small for table reuse

Key Insight

LLMs are memory-bandwidth bound (GEMV with huge OC ≥ 4096) — ternary weight compression directly reduces the bottleneck. CNNs are compute bound (GEMM with moderate OC) — ternary compression helps model size but not compute speed.

Why LUT Doesn't Work for CNNs

How TL2 LUT Works

Precompute a 256-entry lookup table per weight group: for each possible uint8 activation value, store the precomputed dot product. Inference becomes a series of table lookups instead of multiply-accumulate.

Why It Works for LLMs

  • GEMV is memory-bandwidth bound
  • OC ≥ 4096 — massive table reuse across tokens
  • Table build cost amortized over many queries

Why It Fails for CNNs

  • Low OC (64–512) — insufficient reuse
  • GEMM not GEMV — compute bound, not memory bound
  • dpbusd already achieves 32 MACs/cycle
  • Table pollution in L1 cache
Break-even analysis: dpbusd = 32 MACs/cycle vs LUT ≈ 1 lookup/cycle = 4x disadvantage for LUT on CNN workloads.
