Architecture: INT8-Resident Execution with Spatial-Packed dpbusd

Inference Pipeline

Input FP32 → Conv1 FP32 im2col → fused GEMM + BN + ReLU + quantize → uint8 → For each block: uint8 → im2col → repack → 14-OC dpbusd GEMM → fused dequant + ReLU + requant → uint8 → AvgPool → FC → logits

All stages parallelized via persistent omp parallel region. BN fused into conv scale+bias at export time.

What Stays INT8

  • Activations between all layers (uint8, post-ReLU)
  • im2col + repack operate on uint8 bytes
  • dpbusd: u8 × i8 → i32 accumulation
  • Static quantization scales (calibrated offline)
  • Shortcut convolutions also INT8 via dpbusd

Where FP32 Appears

  • First convolution (Conv1) — FP32 input, FP32 GEMM
  • GEMM epilogue (i32 → fp32 → scale → u8)
  • Residual addition (dequant + add + ReLU + requant)

Key Innovation: Spatial-Packed dpbusd

Before: Reduction-Major dpbusd

  • 32 MACs along reduction dimension
  • 1 spatial position per instruction
  • Requires horizontal reduce after each group
  • Port 6 utilization: 55%
  • Whole-inference CPI: 0.294

After: Spatial-Packed dpbusd

  • 8 spatial × 4 reduction per instruction
  • No horizontal reduce needed
  • GEMM kernel CPI: 0.179 (VTune-verified)

Combined with 14-OC Unrolled Tiling

Each dpbusd instruction processes 8 spatial positions across 4 reduction elements, with 14 output channels unrolled in the outer loop. 14 accumulators + 1 activation + 1 weight = 16 YMM registers (full file). This eliminates horizontal reduction and maximizes instruction throughput.

Ternary vs INT8: Where Ternary Wins and Loses

Where Ternary Wins

  • Model size: 3.56 MB vs 11.23 MB (3.2x smaller)
  • Memory bandwidth: Smaller weights = less data to move
  • Edge deployment: Fits in tighter memory budgets

Where Ternary Loses

  • Compute throughput: dpbusd processes any INT8 value at the same speed — ternary gets no compute advantage
  • Sparsity: Only 14% group-level sparsity, not enough for skip-based speedup
  • LUT: vpshufb LUT is 1.8–4.0x slower than dpbusd on CNN layers (measured)

Key Insight

LLMs are memory-bandwidth bound (GEMV with huge OC ≥ 4096) — ternary weight compression directly reduces the bottleneck. CNNs are compute bound (GEMM with moderate OC 64–512) — ternary compression helps model size but not compute speed. The hardware multiplier is "free" — dpbusd's 32 MACs/instruction run at the same throughput regardless of weight values.

What Didn't Work (and Why)

Ternary-Specific Compute Tricks

  • INT16 add/sub: Fewer MACs/instruction than dpbusd
  • sign_epi16: Requires unpacking, negation, extra shuffle
  • Sparse / zero skipping: Branch misprediction overhead exceeds savings (only 14% group-level sparsity)
  • FP32 add/sub: Wider registers but half the throughput of INT8 dpbusd

Three Attempts to Eliminate im2col+repack

  • Fused implicit GEMM: 58ms (address computation overhead)
  • Fused im2col → packed: 4.71ms (marginal, 3x more instructions)
  • Unrolled 9-pos: 4.58ms (2.2x more instructions)

8-OC Register Tiling (Early Attempt)

Too few OCs per tile — 2.7× more activation re-reads offset register benefit. Later solved with 14-OC unrolled tiling.

Lesson: every "obvious" optimization must be benchmarked. Fused im2col, sparse skipping, and small register tiles all looked good on paper but measured worse.

4-bit sign+add

4-bit nibble-packed dpbusd

Packing two 4-bit values into one byte halves activation storage, but the unpack/repack overhead inflated im2col from 2.7B to 27.9B instructions. Accuracy also drops by 0.82pp below the 94.46% threshold.

dpbusd processes 32 MACs per instruction in INT8. Any scheme that cannot use dpbusd natively will be slower.

MLAS Analysis: Closing the ONNX Gap

Key Discovery: ONNX Runtime Uses MLAS, Not oneDNN

oneDNN has no JIT INT8 kernels for AVX-VNNI (Raptor Lake) — falls back to reference C code (7,400× slower). ONNX Runtime uses MLAS (Microsoft Linear Algebra Subroutines) with hand-optimized AVX-VNNI kernels.

MLAS Inner Loop (6-OC × 16-N)

vmovdqu (%rdx), ymm0 ; 8 u8 activations [0:7] vmovdqu 0x20(%rdx), ymm1 ; 8 u8 activations [8:15] vpbroadcastd (%r10), ymm2 ; OC0: weight vpdpbusds ymm0, ymm2, ymm4 ; OC0 acc[0:7] vpdpbusds ymm1, ymm2, ymm5 ; OC0 acc[8:15] ; ... OC1→ymm6/7, OC2→ymm8/9 ; OC3→ymm10/11, OC4→ymm12/13 ; OC5→ymm14/15

Why LUT Doesn't Work for CNNs

How TL2 LUT Works

Precompute a 256-entry lookup table per weight group: for each possible uint8 activation value, store the precomputed dot product. Inference becomes a series of table lookups (vpshufb nibble decomposition) instead of multiply-accumulate.

Why It Works for LLMs

  • GEMV is memory-bandwidth bound
  • OC ≥ 4096 — massive table reuse across tokens
  • Table build cost amortized over many queries

Why It Fails for CNNs

  • Low OC (64–512) — insufficient reuse
  • GEMM not GEMV — compute bound, not memory bound
  • Per-weight LUT table thrashes L1 at high K (layer4: 512×4608 tables)

Root cause: instruction count. vpshufb has better IPC and far better cache behavior, but generates 2.76× more instructions. dpbusd compresses 32 MACs into 1 instruction (8 spatial × 4 reduction). vpshufb needs ~15 instructions for 32 MACs (load + 2 nibble splits + 2 table broadcasts + 2 shuffles + 4 sign-extends + 4 adds). The hardware multiplier is free — replacing it with lookups just adds work.

LUT loses on instruction count, not cache or IPC. dpbusd = 3 instructions per 32 MACs; vpshufb = 15 instructions per 32 MACs. Better cache behavior cannot compensate for 2.76× more instructions.

A12

Why No Other Instruction Matches dpbusd

The Core Question

Ternary weights are {-1, 0, +1} — every multiply is trivial. Can we skip the multiplier and use simpler add/sub/lookup instructions instead? No. We benchmarked every alternative on the AVX2+VNNI ISA.

Instruction Operation Useful Ops / Instr Throughput Why Slower
dpbusd (VNNI) u8 × i8 → i32 accumulate 32 MACs 1 cycle —
maddubs_epi16 u8 × i8 → i16 (pre-VNNI) 16 MACs 1 cycle Half throughput + needs horizontal add to i32
add_epi16 (ternary add/sub) Conditional add or subtract 16 adds 1 cycle Half throughput + branch/sign logic overhead
sign_epi16 (branchless) Negate by sign mask 16 ops 1 cycle Same port saturation + unpack overhead
add_epi32 / sub_epi32 INT32 add/sub per lane 8 ops 1 cycle 4× fewer ops per instruction
add_ps (FP32 add/sub) FP32 add per lane 8 ops 1 cycle 4× fewer ops + wider data (4 bytes vs 1)
vpshufb (LUT) 16-entry nibble lookup 32 lookups 1 cycle Needs nibble decomposition → 2.76× more total instructions

Why dpbusd Wins

  • 32 MACs in 1 instruction, 1-cycle throughput
  • The hardware multiplier costs zero extra cycles — it is fused into the datapath
  • Any scheme that avoids the multiply also gives up dpbusd's 32-wide accumulate
  • You cannot beat "free multiply + 32 ops/cycle" by cleverly avoiding a free operation

Measured Evidence (Experiments 3–6, 18)

  • INT16 add/sub: 17.5ms vs dpbusd 9ms
  • FP32 add/sub: 81ms vs dpbusd 9ms
  • 4-bit sign+add: 32.9ms vs dpbusd 4.7ms (8.2× more instructions, VTune verified)
  • vpshufb LUT: 1.8–4.0× slower across all layers
The multiplier is free in silicon. Ternary's "trivial multiply" provides no compute advantage on VNNI hardware — dpbusd processes {-1,0,+1} at the same 32 MACs/cycle as arbitrary INT8. The only wider option would be AVX-512 dpbusd (64 MACs/instr), but that's the same instruction, just wider.
